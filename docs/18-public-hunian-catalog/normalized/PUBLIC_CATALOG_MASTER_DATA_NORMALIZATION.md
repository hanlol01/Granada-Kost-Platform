## 11. Recommendation for Next Step

### M18B — Backend / Public Catalog API / Data Model

M18B should:

1. **Use this normalization document as the content source** — the data contract in Section 9 defines the exact shape of catalog items.
2. **Expose public-safe catalog items only** — all content must pass the safety checklist in Section 10; backend is the enforcement point (per M18A §9).
3. **Keep existing M16 availability counts** — `availabilityCount` maps to M16D `availableCount` aggregation; no new inventory logic.
4. **Support slug-based detail lookup** — `GET /api/v1/public/catalog/:slug` returning the catalog item contract.
5. **Support listing endpoint** — `GET /api/v1/public/catalog` with optional `category` and `gender` filters.
6. **Keep booking lead defaults compatible with M17** — `bookingLeadDefaults` maps 1:1 to `POST /api/v1/public/booking-leads` payload; no new lead fields.
7. **Remain additive only** — no breaking changes to existing M16/M17 APIs, models, or database schema beyond additive catalog data.
8. **Not implement payment booking** — consistent with M18A freeze §8.
9. **Handle missing content gracefully** — nullable fields (`priceFromMonthly`, `priceFromYearly`, `gallery`, `faq`) return null/empty and the frontend hides or shows placeholder copy.
10. **Store catalog content** — decide storage approach (database table, config/seed file, or hybrid) for the catalog item content. Seed with normalized content from this document.
11. **Implement rate limiting** — public catalog endpoints should be rate-limited consistent with M16D posture.
